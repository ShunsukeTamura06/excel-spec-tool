# xlblueprint ビジョン / ロードマップ

> ステータス: **採用 (adopted, 2026-07-01)**。技術スパイク（§6）で3前提の実現可能性を
> 確認した上で採用を決定。`docs/SPEC.ja.md` §1.1・§1.3・§9 に本書の内容を反映済み
> （本書末尾「SPEC への反映」参照）。最初の利用者は**社内の直したい人（S2 ミニ）**に決定
> （§6.4）。英語版 `VISION.md` は追って整備する。

## 0. 上位絶対条件（本書・SPEC より上位）

**目的**: JTC 等に大量に存在する老朽化した Excel マクロツールを、**解析でき、安全に保守・改修できるようにする**こと。

**背景**: Claude Code 等のコーディングエージェントは既にファイルの解析・改修を行えるが、
Excel は VBA・セル・シート・オブジェクト・名前定義・数式・参照といった Excel 固有の
複雑な仕組みを持つため、汎用エージェントだけでは確実な解析と安全な改修を実現できない。

**帰結**: 本プロダクトが用意すべきものは「Excel を解析・改修するエージェント」そのもの
ではなく、**Excel 専用の、確実な解析と安全な改修を実現する道具（能力レイヤー）**。それを
使う主体（人間の担当者向け画面、コーディングエージェント等）は、この道具の上に乗る「口」
であり、道具ができたあとに実装する / 既存のものを流用する。

この節は本書・SPEC.ja.md の他の記述より優先する。矛盾する記述があれば本節を正とし、
該当箇所を改定すること。

### 0.1 帰結: 投資の配分

| レイヤー | 位置づけ | 投資方針 |
|---|---|---|
| core (解析: 抽出・設計書・参照インデックス・リスク解析・正規化 diff) | 道具の本体 | 主投資対象 |
| core (改修: 安全な書き込みプリミティブ + 検証) | 道具の本体 | 主投資対象。§4.3 の EditPlan 方式で実装 |
| Web UI + チャット (backend/frontend) | 道具を使う「口」の一つ。VBA が書けない社内担当者向け | 既存を流用・簡素化。新規の丁寧な誘導機能は追加しない |
| MCP サーバー / CLI | 道具を使う「口」の一つ。コーディングエージェント向け | 新規だが薄いラッパー。core の関数をそのまま公開する |

### 0.2 やってはいけないこと

- 「安全な改修」の確実性をプロンプト（LLM への指示文）で作ろうとすること。確実性は
  決定的なコード（core の検証ロジック）に置く。プロンプトは道具が欠けている部分の
  暫定的な穴埋めであり、恒常的な設計要素にしない
- 特定の UI・特定のエージェントに最適化した機能を core に持ち込むこと（層の逆流）
- 想定ユーザーを画面ごとに個別最適するのではなく、「口」ごとに自然に分離させる
  （担当者向け画面＝承認のみ、エージェント向け＝core をフルに使う）

## 1. なぜ（課題）

現行 SPEC §1.1 のフローは **5. 「ユーザーは指示に従って手作業で Excel を改修する」** で終わる。
しかし想定ユーザー（VBA に不慣れな保守担当・一般ユーザー）は:

- LLM の修正手順を手作業で適用する過程で、**不要な削除や改変を入れて新たなバグを埋め込む**
- LLM 側は「修正方法」を答えるだけで、ユーザーが完全に指示どおり直すこと・テストを行うことを前提にしている

つまり製品は「**最後の一歩（安全に適用し、検証する）を、それができない人に丸投げしている**」。
この不一致を埋めることが本ビジョンの目的。

## 2. 北極星（最終目標）

ユーザーが LLM に指示すれば、**設計 → 影響範囲確認 → 修正 → テスト → バージョン管理**までを
（適切な承認を挟みつつ）自律的に行い、ユーザーは修正済み Excel を使うだけでよい状態。

ただし「Excel IDE」という言葉が混ぜている二つの製品像 — 開発環境（エンジニア向け）と
ゼロタッチ自律（非開発者向け）— は方向が逆。**コードを読めない人に刺さるのは IDE ではなく、
“証拠（差分・テスト結果）とロールバックで信頼できる、ガード付き自律エージェント”**。

## 3. 出荷アイデンティティの段階進化（重要）

北極星は保持しつつ、最初から自律修正を狙わない。**最初に出すのは「変更の安全ゲート / 検収」**。

| 段階 | 製品の正体 | 誰が変更を作るか | 価値 |
|---|---|---|---|
| **S1: Excel 納品受入ゲート** | ファイルを開く（「前」を自動スナップショット）→「後」が決まると、差分・危険箇所・再現テスト・ロールバックを出す | 人間 / 既存ベンダー / ツール | 「直した Excel を安全に検収できる」。責任境界が明確、自律化なしで価値 |
| **S2: 限定自動リファクタ** | 安全パターンのみ自動適用（名前定義修正・固定参照置換・数式範囲拡張）。**数式先** | ツール（狭い範囲） | 「壊さない自動修正」をテスト付きで |
| **S3: 自律修正（広域）** | VBA を含む修正を VBIDE COM で適用、リスクスコア連動で承認/自動 | ツール（広い範囲） | 北極星に接近 |

S1 が後段に必要な基盤（差分・テスト・リスク・監査）をそのまま育てる。

**S1 の位置づけ（重要）**:
- 価値の呼び名は「AI 改修支援」でなく **「Excel 納品受入ゲート / 回帰テスト / 監査証跡」**。
- **“ただの差分レポート”は弱い。** 最初から **「受入ゲート＋自動回帰テスト雛形＋差戻しレポート」を一式**で出す。
- 主戦場ペルソナ＝**④検収**と**②エンジニア**。**①初心者は S1 単体では弱い**（赤判定を直す力がないため、本領は S2 以降）。売れる条件＝外注受入 / 事故コストの高い月次・四半期 Excel / 監査・内部統制 / 属人ツールの引継ぎ。
- **S1 は LLM 依存を最小化**：diff・テスト・危険判定は決定的処理に寄せ、LLM は説明文生成と探索補助に限定（閉鎖網でのモデル品質保証問題を回避）。エージェント基盤は S2/S3 から。
- **入力の手間は最小（重要）**：ユーザーが改修前後の2バージョンを準備する必要はない。「前」はファイルを開いた時点で自動スナップショット、「後」は (a) ツールが作る（S2）/ (b) ユーザーが編集すると自動記録 / (c) 外注が納品（このときだけ自然に2版）。製品像は「2ファイルを比較する箱」でなく **「1ファイルを開くと変更を自動記録・チェックする相棒」**。

## 4. 設計の核

### 4.1 検証ループ（堀）
- Windows + Excel クライアントにオンプレ実行し、**Excel COM で実際に再計算・マクロ実行**して挙動を観測。静的解析（既存 `reference_index` / `risk_analyzer`）＋動的検証。
- **キャラクタライゼーション（現状挙動の記録）は合否ゲートにしない**。現状が正しい保証はなく、合否化すると「意図した変更まで赤」になり信頼を失う。→ **blast radius ハイライター**として、何が変わったかを提示し人間が意図/非意図を仕分ける材料にする。
- **差分テスト**: 原本と修正版に同一入力を与え「意図したセル/挙動だけが変わったか」を提示。値だけでなく表示形式・条件付き書式・データ検証・名前定義・印刷範囲・マクロ副作用も対象（少差分＝安全とは限らない）。
- **決定性**: 揮発性関数の固定化、外部リンク/接続はフィクスチャ録画、実行メタデータ（Excelバージョン/ロケール/計算モード/TZ）を記録。**外部リンク未解決は日常状態なので「環境不備で診断不能」にしない**運用が要る。
  - **RTD 系アドイン関数（Bloomberg BDP/BDH 等のリアルタイム市場データ取得関数）はエッジケースでなく主要な非決定性の発生源**（§6 スパイク結果で実測）。金融系の業務 Excel では頻出するため、P1 の時点で「該当セルをスナップショット時に値へ凍結する」または「比較対象から除外する」対処を組み込む必要がある。後回しにすると差分テストがほぼ常に赤判定になり、S1 の信頼性が成立しない。
- 隔離コピーで実行し、未信頼 VBA 実行はサンドボックス分離（別ユーザー/VM）。

### 4.2 信頼モデル（“別人の承認”でなく“証拠による自己確信”）
- 現場では「直す人＝確かめる人＝確定する人」がほぼ同一で、**「承認だけする専任者」は実在しない**。よって責任ロールの分離を前提にしない。
- 代わりに、**ツールの証拠（差分・テスト・ロールバック）が、従来は熟練者レビューが必要だった部分を肩代わりする**。これが中心価値。承認は「別人の承認」でなく、**操作者自身が証拠を見て確信して確定するチェックポイント**。
- 承認が本当に分かれるのは限定ケースのみ、かつ**任意（強制しない）**: ①外注の納品物を受け取る検収（S1の本命ユースケース）、②規制業種・大規模で監査分掌が要る場合。
- **初心者ゼロタッチは後回し**（誤修正時の責任所在が曖昧になるため）。
- **監査証跡**は「リアルタイムの承認役を置くため」ではなく**事後の説明責任のため**に残す。
- ロールバックは万能でない（修正済み Excel で作った成果物・他システム登録・会計処理は戻せない）と明示。

### 4.3 変更の適用と差分
- 数式は文字列置換でなく **AST 化して安全に変更**。出力/中間/入力セルを分類。
- 削除は静的参照ゼロ＋**VBA 文字列参照・INDIRECT・名前定義・フォーム/グラフ/ピボット・手運用**まで考慮（静的ゼロ＝安全ではない）。
- VBA 書き戻しは **VBIDE COM**（`vbaProject.bin` 直接編集は避ける）。**署名失効・保護ブック・VBIDE 信頼設定(GPO)** への対応が必須。
- 差分は **構造 diff**（セル数式/VBAプロシージャ/名前定義/入力規則 単位）。xlsm は保存ノイズ（calcChain/sharedStrings/style/rel id/pivot cache）が多く、**ノイズ除去が本体**。裏で正規化して git 管理すれば履歴/blame が付く。
- **改ざん検知付き監査証跡**（指示文/モデル/計画/差分/テスト結果/承認者/時刻）。

### 4.4 配布
- **コア＝ローカルエンジン**（既存 core/backend を流用）＋ 複数の「口」を用意する（§0）。
- 「口」はペルソナで分ける:
  - **①初心者・④検収 → 既存 Nuxt Web を流用**：受入/差戻し・「このセルだけ確認」など
    承認中心の画面。丁寧な誘導機能をここで作り込まない（§0.2）。
  - **②エンジニア・コーディングエージェント → CLI（先行）→ MCP サーバー（後続）**
    （いずれも新規だが薄いラッパー）。core の解析系関数（抽出・参照検索・リスク一覧等）
    と改修系関数（propose_edit_plan・apply_edit_plan）をそのまま公開する。Claude Code 等の
    汎用エージェントが直接 Excel を扱えるようにする本来の目的（§0）に対応する入口。
    閉鎖網では MCP クライアント自体が導入できない可能性が高いため、まず CLI を
    安定 API として作り、MCP は CLI/core を薄く包む後続とする（§6.7）。
    VS Code 拡張は CLI/MCP でエンジニア需要をカバーできる場合は不要（借りるものを
    増やさない）。
- **Excel アドイン(VSTO) は後段**（.NET/署名/GPO/EDR/更新で詰まりやすく、多面展開は運用コストが重い）。

### 4.5 作らない設計（build vs borrow）
我々は IDE・エージェント・VCS・diff ビューア・数式エンジン・COM ラッパーを**作らない**。汎用基盤に挿す **Excel 専用層**だけを作る。

| 領域 | 借りる | 我々が足す |
|---|---|---|
| 版管理 | **git 本体** | xlsx→正規化テキスト/抽出VBA/数式/名前定義/シート構造 の安定変換層 |
| エンジニアUI | **VS Code**（diff/SCM/webview/拡張ホスト） | Excel 固有ビュー（設計書・依存グラフ・検収レポート） |
| テスト基盤 | **pytest**（fixture/レポート） | 入力投入→再計算→出力比較→証跡保存 のハーネス |
| Excel 操作 | **xlwings**（インスタンス管理/計算） ＋ **pywin32**（VBIDE/低レベル） | プロセス分離・タイムアウト・強制kill・失敗復旧 |
| VBA 抽出/危険語 | **oletools(olevba)** | 業務VBAの意味解析・依存解決・リファクタ安全性 |
| 数式再計算（補助） | formulas/pycel/xlcalculator（Excel無しの高速テスト用） | — 最終判定は COM 上の本物の Excel を真値に |
| エージェント基盤（S2/S3） | **opencode / Goose**（OpenAI互換・ツール拡張）or Cline（VS Code/承認体験） | Excel 固有ツール（read_sheet / get_dependencies / apply_formula_edit / run_test …）。※Continue は EOL のため除外 |
| コーディングエージェント連携 | **Claude Code 等の汎用エージェント本体**（MCP クライアントとして接続するだけ） | **MCP サーバー**（core の解析/改修関数を薄くラップして公開するだけ。エージェントのロジックは持たない） |

**自作すべき“真の核（堀）”は3つだけ**:
1. **Excel 正規化・意味 diff 層**（安定IDで抽出しノイズを消して差分化）
2. **Excel 回帰テスト生成/実行/証跡層**（入力シナリオ・観測セル・再計算・マクロ実行・許容差・失敗ログ・ロールバック）
3. **変更リスク判定ルール**（INDIRECT/OFFSET・イベントマクロ・外部リンク・削除・名前定義変更・参照切れ・保護/アドイン依存 を業務 Excel 向けにスコアリング）

## 5. ロードマップ（改訂版）

1. **P0（済）**: 構造抽出・設計書・参照インデックス・リスク解析・改修チャット（助言）
2. **P1: 安全ゲート / 検収レポート** — 改修前後＋意図から、構造差分・依存影響・危険箇所・実行ログ・ロールバック用コピーを出す。*修正はしない。* ← 最初の出荷価値
3. **P2: 再現テストの半自動生成** — 主要セル/操作から差分テストを生成（合否ゲートでなく blast radius 提示）
4. **P3: 限定自動リファクタ（数式先・安全パターン限定）** — AST ベース、承認チェックポイント
5. **P4: VBA 自動修正（VBIDE）＋マクロ実行テスト**
6. **P5: 署名/監査/Git 連携 → リスクスコア連動の限定ゼロタッチ**

## 6. 先に潰すべき前提（スパイク）

実装前に小さく検証する。失敗したら設計が変わる:

1. **現状挙動を“再現可能なテスト”に落とせるか（最大の壁）** — 揮発関数・外部リンク・イベントマクロ・ファイルパス依存・ユーザー操作依存を、入力固定して決定的に再現できるか。COM が動いてもここが不能なら検証は無価値。
2. **企業クライアント実機で Excel COM / VBIDE が安定動作するか** — GPO の VBIDE 信頼無効、保護ビュー、秘密度ラベル、EDR、モーダルダイアログで無人実行が止まらないか（運用隔離・タイムアウト・強制kill で抑えられるかも併せて）。
3. **xlsm 正規化 diff のノイズ除去**（calcChain/sharedStrings/style/rel id/pivot cache）が実用精度に届くか。

### 6.1 スパイク結果（2026-07-01、実業務 .xlsm 1本・社内 S 端末で実行）

`spikes/com_probe/probe.py`（PR #20）を実機で実行し、上記 1・2 について一次判定を得た。

| 診断項目 | 結果 | 所見 |
|---|---|---|
| ヘッドレス起動・ワークブックオープン | ✓ ok | ダイアログ抑制で無人実行できた |
| ベーススナップショット | ✓ ok | 全セルの計算結果取得は問題なし |
| 2回再計算の決定性 | 一部 NG（28/数千セル） | **全件を辿ると Bloomberg `BDP` に到達**。原因不明の構造的な非決定性ではなく、外部データフィード起因と特定できた |
| VBIDE 経由の VBA アクセス | ✓ ok | GPO で `AccessVBOM` は塞がれていなかった。**S3（VBA 自律修正）の最大の懸念が解消** |

**結論**: 1・2 とも致命的な失敗はなし。ただし **RTD 系アドイン関数（Bloomberg BDP/BDH 等）による非決定性は「稀なエッジケース」ではなく「金融系業務 Excel では常時発生する主要因」** であることが判明（社内ヒアリングで BDP/BDH は日常的に使用と確認）。→ §4.1 に対処方針を追記。

**限界**: サンプルは実務ファイル1本のみ。ファイル間で非決定性の傾向（RTD 以外の原因が出るか、件数の桁が変わるか）が変わる可能性があるため、本格実装前に複数ファイルでの追加確認が望ましい。

### 6.2 スパイク結果（2026-07-01、xlsx 正規化 diff ノイズ除去）

`spikes/xlsx_diff_noise/probe.py` で上記 3 を検証した。**Windows/Excel 不要、このPC単体で完結**。
`scripts/make_sample.py` のサンプルをベースに、(A) 意味的な変更なしで保存し直しただけのファイルと、
(B) 1セルの数式だけを書き換えたファイルを作り、それぞれ「生 zip パーツのバイト単位 diff」と
「既存の `core.extractors.cells.extract_cells_to_sqlite`（セル単位の正規化抽出）経由の diff」を比較した。

| ケース | 生XML diff | 正規化抽出 diff |
|---|---|---|
| (A) 意味的な変更なしで保存し直しただけ | `styles.xml` / 無関係な `sheet*.xml` / `chart*.xml` など**5パーツが変化**（ノイズ） | **0件**（一致） |
| (B) 1セルの数式のみ書き換え | (A)のノイズ一式 + 変更した `sheet*.xml` の**計6パーツが変化** | **変更した1セルのみ**を正確に検出 |

**結論**: **仮説成立（`verdict.hypothesis_confirmed = true`）**。xlsx 専用の正規化 diff レイヤーを新規に作る
必要はなく、**既存の `extract_cells_to_sqlite`（元々は LLM 向けの生データ提供用に作った関数）をそのまま
比較の基盤に転用すれば、保存ノイズは素通りできる**。§4.5 の build-vs-borrow でいう「借りる」対象が
1つ増えた形（自作コストがさらに下がった）。

**限界（当初は未検証。§6.3 で実機確認済み）**:
- 検証したのは **openpyxl による保存ノイズ**のみだった。VISION が本来懸念していた **Excel 本体で開いて
  保存し直したときの**ノイズ（calcChain 再構築・sharedStrings の頻度順並び替え・.rels id の大幅な振り直し
  等）は openpyxl より激しい可能性があった → §6.3 で実機検証済み。
- `extract_cells_to_sqlite` は**セルの値・数式・表示形式・結合範囲のみ**を見る。名前付き範囲・条件付き
  書式・データ検証・グラフ・ピボットテーブル・印刷範囲・マクロの副作用など、§4.1 が挙げる他の差分対象
  は `core.extractors.workbook.extract_workbook()` 側の構造化データを別途比較する必要があり未検証。
- 大規模ファイルでの性能（ワークブックを2回ロードする実装）は未計測。

### 6.3 スパイク結果（2026-07-01、実業務 .xlsm・本物の Excel での保存し直しノイズ）

`spikes/com_probe/probe.py` に追加した `excel_resave_noise` ステップ（PR #27）を、§6.1 と同じ実業務
.xlsm で実機実行した。ワークブックを `SaveCopyAs` で保存し直すだけ（編集なし）にして、(a) 生 zip パーツの
バイト単位 diff と (b) 計算値ベースの diff（§6.1 で既知の Bloomberg 由来 28 セルを差し引いたもの）を取得。

| 項目 | 結果 |
|---|---|
| `raw_xml_parts_changed_count` | **12 パーツ**（openpyxl の 5〜6 パーツより多い＝実機の方がノイズは激しいという事前の懸念どおり） |
| `unexplained_by_known_volatility_count` | **0 件** |

**結論**: **本物の Excel でも仮説成立**。保存し直しただけで生XMLは openpyxl 以上に広範囲が変化するが、
計算値ベースで見ると「既知の Bloomberg 由来セル以外に変化したセルは1つもない」。§6.2 で示した

成立する**ことが確認できた。§6 の3前提（1〜3）はこれで全て実機/実証込みで一次検証を終えた。

**残る限界**: サンプルは実務ファイル1本のみ。計算値ベースの比較は §4.1 で挙げた他の差分対象（名前付き
範囲・条件付き書式・グラフ・ピボット等）をカバーしない（未検証のまま）。複数ファイルでの追加確認は
引き続き望ましい。

### 6.5 P1 増分1完了後の優先度決定（2026-07-01）

P1 増分1（`core.workbook_diff` + `GET /diff` + `/diff` ページ、§本文末尾コミット履歴参照）を
実装し、ブラウザで実際に動く状態にした。ここでユーザーから重要な指摘があった:
**「毎回ユーザーが手動で改修し、前後2ファイルをアップロードして比較する」という運用は、
そもそも北極星（§2）が目指す「ユーザーは指示するだけで、ツールが影響範囲を見ながら自動修正し
完成品を返す」から外れている**。

指摘は正しい。増分1で作った差分エンジンは S1（受入ゲート、人間が直したものを検収する）の
土台として単体でも価値があるが、**それ単体を磨く（前後バージョンの自動紐付けUI等）よりも、
S2（限定自動リファクタ：ツール自身が安全なパターンに限定して修正を適用する）に進む方を
優先する**と決定した。

理由: S2 に進んでも増分1の差分エンジンは無駄にならない。**ツールが自分で適用した修正が
意図しない箇所を壊していないかを自己検証する仕組み**として、そのまま転用できる
(§4.3 の「変更の適用と差分」参照)。むしろ人間の手動修正を検収する用途から、ツール自身の
自動修正を検証する用途へと主戦場が移るだけで、エンジン自体の設計は変わらない。

**次の実装対象 = S2 増分1**: チャットで改修を依頼 → 安全なパターン（名前定義修正・固定参照
置換・数式範囲拡張。§3, §4.3）に限定してツールが自動適用 → 増分1の差分エンジンで自己検証 →
結果を提示、というループの最小構成を設計する。スコープの具体化は次のセッションで行う。

### 6.6 S2 増分1 完了 (2026-07-04)

§6.5 で決めた S2 増分1 が完成した (名前定義修正 = PR #35/#37/#38、固定参照置換・
数式範囲拡張 = PR #40)。3パターンとも共通の設計:

- **propose (試算)**: LLM が tool loop から呼べる read-only の影響試算。変更される

- **apply (適用)**: 人間が画面のボタンを押した時だけ backend 経由で実行。元ファイルは
  変更せず**新しいジョブ**として書き出し、増分1の差分エンジンで before/after を比較して

- 数式の書き換えは文字列置換でなく openpyxl formula Tokenizer による参照トークン

**現時点の限界**: 自己検証は構造 diff のみで、COM 再計算による動的検証 (§4.1) は

**次の増分候補** (優先度は最初のパイロット利用のフィードバックで決める):
- A. COM 再計算による動的自己検証 (§4.1 の検証ループ本体。S 端末が必要)
- B. 安全パターンの拡充 (例: 列挿入に伴う参照シフト、シート名変更の追従)

### 6.7 S2 実装方式の転換 + プロダクト正体の再定義（2026-07-06）

**背景**: S2 増分1（名前定義修正）を「パターンごとに専用 core 関数 + 専用 tool + 専用
API ルート + 専用 UI カード」を作る方式で実装した結果、1 パターンに 3 PR を要した
（`core/named_range_fix.py` + `backend/routes/refactor.py` + tool 定義 +
`NamedRangeFixCard.vue`）。次の固定参照置換・数式範囲拡張も同コストが見込まれ、
費用対効果が崩壊する。同時にチャットの system prompt が「コピペ完結の改修手順」を
丁寧に指導するために肥大化し（`backend/routes/chat.py` の `_SYSTEM_INSTRUCTIONS`）、
想定ユーザー（VBA に不慣れな担当者 vs 多少 VBA が分かる人）がブレていた。

この2つの問題を検討する過程で、§0 の上位絶対条件（本セッションでユーザーが明文化）に
立ち返り、プロダクトの正体を再定義した:

**本プロダクトは「Excel を安全に改修するエージェント」ではなく、「確実な解析と安全な
改修を実現する Excel 専用の道具」である。** その道具を使う主体（VBA が書けない社内
担当者向け画面、コーディングエージェント）は道具の上に乗る「口」であり、道具本体に
対して二次的。

この再定義から、S2 の実装方式を以下のように転換する（旧方式である「パターンごとの
専用エンジン」は今後追加しない）:

1. **core**: `EditPlan`（型付き編集操作のリスト: `set_named_range` /
   `set_cell_formula` / `set_cell_value`、将来 `replace_vba_procedure` via
   VBIDE COM）と、op ごとの precondition/postcondition を決定的に検証する
   policy 層を導入する。名前定義修正 (`core/named_range_fix.py`) はこの op の
   1パターンとして吸収する
2. **LLM tool / API / UI**: パターンごとに増やすのをやめ、`propose_edit_plan`
   （dry-run: 既存 `workbook_diff` で差分・blast radius 試算）/
   `POST /jobs/{id}/apply-edit-plan`（新ジョブ作成 → フル抽出 → diff 自己検証）/
   汎用 `EditPlanCard` の1本に集約する
3. **数式編集の安全性**: LLM は「見た目は正しいが意味が違う式」を作りうる
   （相対/絶対参照、structured reference、`@` 暗黙交差、スピル等）ため、当面は
   任意の式生成を許可せず、「参照先の付け替え」「範囲拡張」のような機械的変換に
   限定する。任意式の解放は COM 再計算テスト（P2/P4）が入ってから
4. **想定ユーザー**: §1.2 の「VBA に不慣れな担当者」向け丁寧誘導（コピペ完結手順の
   散文指導）は、ツールが構造的に適用できる操作については不要になるため縮小する。
   ツールがまだ対応しない操作（当面 VBA 全般）にのみ暫定的に残す。「対象ユーザーを
   どちらかに絞る」問題は、口を分けることで解消する: 担当者向け画面は承認のみ、
   エージェント向け MCP は道具をフルに使う
5. **配布**: §4.4 に MCP サーバー / CLI を「口」として追加する。core の解析系関数
   （get_cells_range 相当・lookup_references・list_analysis_risks 等）と改修系
   関数（propose_edit_plan・apply_edit_plan）を薄くラップする。Web UI・チャットへの
   新規投資より優先度を上げてよい（早期にドッグフーディングでき、道具自体の設計
   欠陥を早く見つけられるため）※優先順位は §6.8 のレビューで修正済み

この決定は §0 を反映したものであり、以降の §3〜§5 のロードマップ・設計判断はこれを
前提に更新する。

### 6.8 ストーリーレビューによる優先順位の修正（2026-07-15）

§6.7 の転換ストーリー（田中さん=承認のみ / MCP が主要入口）を、実装前に批判的に
レビューした（Claude + codex のクロスチェック）。骨格（§0「core が本体、UI/
エージェントは口」）は維持するが、以下の3点の弱点が見つかり、優先順位を修正する。

**弱点1: 語彙ギャップ**。非開発者は「名前定義の参照先を変えたい」とは言えず、
「税率が変わったので直したい」としか言えない。難所は「安全な適用」ではなく
**業務語彙 → EditPlan 操作への翻訳**（LLM の仕事として残る）。承認画面の証拠も
op 語彙（名前定義がどう変わる）でなく**業務語彙（このセルの計算結果がこう変わる）**
で見せる必要がある。「承認だけで完結」はこの翻訳が当たる場合に限られる。

**弱点2: 閉鎖網での MCP 前提が弱い**。JTC 閉鎖網では Claude Code 等の MCP
クライアント自体が導入できない可能性が高い（ネットワーク・Node・セキュリティ審査）。
一方 Python は開発端末にある。→ **エージェント向けの口は CLI を先に安定 API として
作り、MCP サーバーは後から CLI/core を薄く包む**（MCP は OSS 公開時の訴求が主）。

**弱点3（最大の急所）: 安全 op の需要カバー率が未測定**。実際の改修依頼の大半が
VBA 改修なら、名前定義・数式・セル値の op をいくら整備しても空振りし、
費用対効果崩壊（§6.7 の背景）が再発する。→ **実装前に、実在する改修依頼を
20〜50 件集めて分類し、安全 op でカバーできる割合を測る**（社内で安価に実施可能）。

**採用した代替視点（検証特化）**: 変更は誰が作ってもよい（人・openpyxl スクリプト・
エージェント）とし、xlblueprint は**前後スナップショット + 正規化 diff + 波及範囲 +
リスク提示を確実にやる**ことに主軸を置く。これは元の S1（安全ゲート）そのものであり、
`core/workbook_diff` として大部分が実装済み。エージェントは openpyxl で自由に
書けるので、彼らに足りないのは書き込み手段ではなく**「壊していないことの確認手段」**。
ただし非開発者の「承認だけ」体験には EditPlan が必要なため捨てず、
**需要調査で上位に来た 1〜2 op に絞って**実装する。

**修正後の優先順位**:

1. **需要調査**: 実在の改修依頼を収集・分類し、安全 op のカバー率を測る（S2 の
   価値仮説の合否判定。ここで VBA 改修が多数派なら S2 の価値仮説を作り直す）。
   手順・分類軸は [docs/s2_demand_survey.ja.md](./s2_demand_survey.ja.md) 参照
2. **検証エンベロープ + CLI**: 解析（抽出・参照検索・リスク一覧）と検証
   （スナップショット・正規化 diff・波及範囲）を CLI として安定化。閉鎖網でも
   動き、エージェントからも使え、ドッグフーディングを即開始できる
3. **EditPlan**: 需要調査で実証された op のみ実装。承認 UI は業務語彙の証拠で提示
4. **MCP サーバー**: CLI が固まった後の薄いラッパー（OSS 訴求用）

### 6.9 検証コントロールプレーン増分1（2026-07-16）

プロジェクトの役割を次の一文で固定した。

> OfficeCLI が「Office を変更できる手」なら、xlblueprint は「変更内容を理解し、
> 壊していないことを証明する頭と検査装置」。

この役割分担をコード上の境界にした。変更意図を `MutationPlan`、書き換え実装を
交換可能な `MutationProvider`、安全判定を期待差分と実差分の厳密照合policyとして分離した。
既存openpyxl修正はこの経路に統合し、OfficeCLIは任意の外部プロセスアダプターとして
`.xlsx` の名前定義変更だけを接続した。外部ツールの終了成功は安全性の証拠とせず、変更後の
ファイルをxlblueprint自身がフル再抽出し、計画・両差分・判定を監査レコードに残す。

この増分が証明する範囲は、xlblueprintが抽出できる構造差分が計画どおりであることまで。
Excelの再計算値、イベントマクロ、外部接続などの動的挙動はまだ証明しない。次段階は
同じpolicy chainへExcel COMによる再計算・マクロテスト証拠を追加することであり、
OfficeCLI固有機能をcoreへコピーすることではない。

### MVP 受け入れベンチマーク（S1 の合否基準）
実在 .xlsm を **10本**集め、各2版を作る（**安全変更5 / 危険変更5 / 意図的バグ10** を混入）。S1 で
差分抽出 → 危険検知 → COM 再計算 → 主要セル比較 → マクロ実行 → ロールバック を**1クリック**で回し:

- **検収担当が30分以内に「受入/差戻し」を判断できる**
- **重大バグ検出率 ≥ 80%**、誤検知が検収不能にならない量
- **1ファイル10分以内**、ゾンビ Excel / ハングから自動復旧
- **レポートだけで外注へ差戻し可能**

## 7. 成功指標

「自律修正率」ではなく:
- 危険変更の**検出率**
- **検収時間**の短縮
- **差し戻し / ロールバック**の削減

## 8. ガードレール / 非目標

- **黙って変更しない**（差分・理由・テスト結果を常に提示）
- テストが保証するのは「意図しない挙動変化がないこと」であって「正しさ」ではない
- 閉鎖網 + モデル差し替え可は QA を壊しやすい → **モデル差し替え時の回帰評価セット**を持つ
- ゼロタッチ初心者運用を初期から提供しない
