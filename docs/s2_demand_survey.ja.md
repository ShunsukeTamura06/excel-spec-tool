# S2 需要調査 — 安全 op のカバー率を測る

> 位置づけ: [docs/VISION.ja.md](./VISION.ja.md) §6.7 で決めた優先順位1番目。
> S2（EditPlan による限定自動リファクタ）に投資する前に、実際の改修依頼のうち
> どれだけが安全 op（名前定義・数式参照・セル値の機械的変換）でカバーできるかを
> 実測し、投資の go/no-go を判断する。

## なぜやるか

VISION §6.6 で作ろうとしていた EditPlan（型付き編集操作）は、実際の改修依頼の
大半が VBA ロジック変更のようなカバー対象外パターンだった場合、投資しても
日常的な価値を生まない（§6.7 の最大の指摘）。実装前に安く検証する。

## 重要: 実データは git に入れない

実際の改修依頼には社内の業務用語・数値・システム名が含まれうる。本プロジェクトは
OSS 公開を控えており（CLAUDE.md §9）、**記入済みの調査データは絶対にコミットしない**。

- 記入は `runtime/s2_demand_survey/` 以下に置く（`.gitignore` で除外済み）
- 本書と `docs/fixtures/s2_demand_survey_template.csv` はテンプレートのみで
  実データを含まないため、コミットしてよい

## 進め方

1. `docs/fixtures/s2_demand_survey_template.csv` を `runtime/s2_demand_survey/data.csv`
   にコピーする
2. 実在した改修依頼を思い出せる範囲・聞ける範囲で **20〜50 件** 埋める
   (VISION §6.7 の目安件数)。情報源の例:
   - 自分がこれまで対応した / 頼まれた改修依頼の記憶
   - 同僚・情シスへのヒアリング（「最近このExcelツールで何を直した？」）
   - 依頼が残っている場合はメール・チャット・チケットの履歴
   - 1件も記憶がなければ「典型的にありそうな依頼」を仮説として埋めてもよいが、
     その場合は `根拠` 列に「仮説」と明記し、後で実データに置き換える
3. `uv run python scripts/tally_demand_survey.py runtime/s2_demand_survey/data.csv`
   を実行し、カテゴリ別件数と安全 op カバー率を集計する
4. 結果を見て、EditPlan にどの op を優先実装するか（あるいはそもそも実装するか）
   を判断する

## 分類軸

依頼1件ごとに、該当するカテゴリを1つ選ぶ（複数にまたがる場合は主要なものを選び
`備考` に補足する）。

| コード | カテゴリ | 説明 | 現行の安全 op |
|---|---|---|---|
| A | 名前定義の参照先変更 | 名前付き範囲の参照先セル/範囲を変える | 実装済み (`core/named_range_fix.py`) |
| B | 数式の参照先変更 | 数式の構造は変えず、参照するセル/範囲だけ変える | EditPlan 候補 |
| C | 数式範囲の拡張・縮小 | SUM/SUMIF 等の集計範囲をずらす | EditPlan 候補 |
| D | セル固定値の変更 | 税率・年度・パラメータ等の定数セルを書き換える | EditPlan 候補 |
| E | VBA中の参照先・定数のみの変更 | VBAコード内のシート名・セル番地・定数だけを変える（ロジックは不変） | 未実装 (VBIDE COM 必要、S3以降) |
| F | VBAロジックの変更 | 分岐・計算式・処理順序など、VBAの挙動そのものを変える | 未実装 (S3以降、対応範囲は限定的な見込み) |
| G | シート/列/行の追加・削除等の構造変更 | レイアウト自体を変える | 未実装 (対応予定なし、将来検討) |
| H | 外部接続・ピボット・グラフの設定変更 | Power Query・ピボット・グラフの設定を変える | 未実装 (対応予定なし、将来検討) |
| I | 書式・表示形式のみの変更 | 数値書式・条件付き書式など、値/式は変えない | EditPlan 候補（優先度は要検討） |
| J | その他・複合 | 上記に当てはまらない、または複数カテゴリにまたがる | 個別判断 |

## 記入する列

`docs/fixtures/s2_demand_survey_template.csv` の列定義:

| 列名 | 内容 |
|---|---|
| id | 連番 |
| 依頼の要約 | 依頼された内容を1〜2文で（固有名詞は一般化してよい。例:「Bloomberg」→「外部データ取得関数」） |
| カテゴリ | 上表の A〜J |
| 実際に変更した内容 | わかる範囲で。わからなければ空欄可 |
| 安全opで対応可能か | `Yes` / `No` / `一部` — カテゴリ A〜D・I に該当し、かつ機械的変換で表現できる場合は `Yes` |
| 根拠 | 実体験 / ヒアリング / 仮説、のいずれか |
| 備考 | 補足（複合カテゴリの内訳等） |

## 結果の解釈

集計スクリプトは複数の割合を出す。**判断には「実データのみ・加重集計」の値を使う**こと
（単純集計は「一部」を満額カウントするため楽観的に出る。仮説データが混ざった値も
信頼度が低い）。目安として:

- **割合が低い（おおよそ2〜3割未満）**: EditPlan の広い横展開は費用対効果に見合わない
  可能性が高い。§6.7 の「検証エンベロープ+CLI」を主軸にし、EditPlan は最頻出の
  1〜2 op だけに絞るか、次段階に先送りする
- **割合が高い**: 上位カテゴリから優先的に EditPlan の op として実装する根拠になる

最終判断は割合だけでなく、件数・深刻度（直さないと何が困るか）も合わせて見ること。
本書はあくまで判断材料を揃えるための調査であり、閾値を機械的に適用しない。

### 「安全opで対応可能か」の判定基準（codex レビューで追加）

`Yes` / `一部` / `No` を付けるとき、カテゴリ (A〜D・I) に該当するかだけでなく、
以下の3点も踏まえて判断すること。迷ったら `一部` にする:

- **業務語彙からの特定可否**: 依頼者の言葉（「税率が変わった」等）から、
  対象セル・名前定義・数式を一意に特定できるか。特定に調査が要る場合は `一部`
- **変更範囲の局所性**: 変更が1〜数箇所に閉じるか、広範囲に及ぶか。広範囲なら `一部` / `No`
- **検証可能性**: 適用後に「意図した箇所だけ変わったか」を diff で確認できるか
  （できないなら安全 op として自動適用する意味が薄い）

### バイアスへの注意

記憶だけに頼ると、印象に残りやすい大きな障害・VBA 案件に偏りやすい。
可能な限りメール・チャット・チケット等の履歴を優先し、記憶だけの分は `根拠` 列に
明記した上で件数を分けて見ること（集計スクリプトは実データと仮説を自動で分離する）。
